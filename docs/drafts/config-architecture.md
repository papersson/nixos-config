# Config architecture: dotfiles vs. Nix, single-repo vs. split

Status: draft. Captures current thinking; no decision made.

## The problem

We currently have three places where config lives, and a fourth that's a copy:

| Location | Role | Source of truth? |
|---|---|---|
| `~/dotfiles/` (separate git repo) | Existing dotfiles, also used on macOS | Yes — for shared text files |
| `/etc/nixos/` (flake) | NixOS system + home-manager | Yes — for the Linux laptop |
| `/etc/nixos/home/patrikpersson/claude/` | Copies of `~/dotfiles/claude/*` | No — duplicate, drifts |
| `~/.claude/`, `~/.config/git/config`, etc. | Generated by HM at activation, or written directly | No — outputs |

The duplicate is the obvious smell. The deeper question is whether the dotfiles repo should exist at all once we're on Nix.

## What "best practice" actually means here

There's no single right answer, but the Nix community has converged on patterns. Ranked from most-Nix-native to least:

### Option A — One repo, Nix-everywhere (nix-darwin + HM)

Single flake exposes:
- `nixosConfigurations.t14` — Linux laptop
- `darwinConfigurations.<mac>` — macOS via nix-darwin
- HM is a module under both
- All text configs (CLAUDE.md, helix, nvim) live in the repo as files
- macOS-specific bits gated by `lib.mkIf pkgs.stdenv.isDarwin`
- Brewfile retired in favour of nix-darwin's `homebrew` module (declarative `brew install`)
- `~/dotfiles` repo archived

**Pros:** One source of truth. Full declarative. Same `programs.git`, `programs.ghostty`, etc. on both platforms with platform diff inside the module. Atomic rollback per machine.

**Cons:** Commit to running Nix on macOS. nix-darwin is well-maintained but is a learning curve. Some macOS-only tools (notably anything that wants to live in `/Applications/` or modify system preferences) need nix-darwin idioms. Brewfile → nix-darwin `homebrew.casks = [...]` is mechanical but takes an evening.

This is what Misterio77's starter, Mic92's dotfiles, and most "serious" Nix users do.

### Option B — Two repos, dotfiles is the source-of-truth for text files

Keep both repos, but eliminate duplication:
- `~/dotfiles` stays the source of truth for text-only configs (CLAUDE.md, nvim, helix, statusline.sh)
- NixOS flake imports `~/dotfiles` as a flake input (`flake = false`)
- HM's `home.file."x".source = "${inputs.dotfiles}/path"` reads from the dotfiles repo
- On macOS, existing `install.sh` symlinks still consume the same files
- HM **modules** (git, ghostty, ssh) replace the dotfiles versions on NixOS — meaning some configs exist in two forms (Nix attrs and text) for things HM models well

**Pros:** Minimal disruption. Existing macOS workflow unchanged. NixOS gets type-checked modules for things HM covers.

**Cons:** Two repos still. For things like git config, you'd maintain both `~/dotfiles/base/git/.gitconfig` (consumed on macOS) and `programs.git.settings` in the HM module (NixOS). Drift risk on those.

### Option C — One repo, Nix on Linux, dotfiles-as-input

Variant of B but the dotfiles repo is folded into the NixOS repo as a subdirectory rather than a flake input. macOS keeps using the dotfiles-style approach but pointed at a subdirectory of the unified repo.

**Pros:** One repo to push/pull. No flake-input indirection.

**Cons:** macOS install.sh becomes weirder. Probably worse than A or B.

### Option D — Status quo + sync discipline

Edit in `~/dotfiles`, copy to `/etc/nixos/home/patrikpersson/claude/`, commit both.

**Pros:** Zero migration work today.

**Cons:** Manual sync; drift is a matter of when, not if. This is what we have right now and it's bad.

## My honest take

For someone learning Nix on the new T14 *and* keeping a productive macOS machine, **Option B is the right intermediate step**, and **Option A is the right destination** once nix-darwin feels worth the commitment.

Specifically:
- **Now:** Move to Option B. Add `~/dotfiles` as a flake input, delete the copied files in `/etc/nixos/home/patrikpersson/claude/`, point HM `home.file` at the dotfiles input. One change, big payoff.
- **Later (3–6 months):** When the NixOS side feels settled and you're tired of maintaining git config in two formats, evaluate Option A. The migration is mostly mechanical at that point.

The reason not to jump straight to A: nix-darwin's learning curve is real, and trying to learn nix-darwin while also learning NixOS on the T14 doubles the surface area you're juggling. Sequencing matters.

## Specifics for Option B (the recommended next step)

```nix
# flake.nix
inputs.dotfiles = {
  url = "github:patrikpersson/dotfiles";  # or git+file:///home/.../dotfiles
  flake = false;
};

# home/patrikpersson/default.nix
{ inputs, ... }:
{
  home.file.".claude/CLAUDE.md".source =
    "${inputs.dotfiles}/claude/CLAUDE.md";
  home.file.".claude/statusline.sh" = {
    source = "${inputs.dotfiles}/claude/statusline.sh";
    executable = true;
  };
  # And similarly for helix, nvim, ripgreprc, etc. as they get migrated.
}
```

After this:
- `/etc/nixos/home/patrikpersson/claude/` is deleted
- Edit CLAUDE.md once in `~/dotfiles`, push, then `nix flake update dotfiles && nixos-rebuild switch` on the T14
- macOS keeps consuming `~/dotfiles` the way it does now

## Things to figure out before committing

1. **Is the dotfiles repo pushed to a remote we can use as a flake input URL?** If yes, `github:patrikpersson/dotfiles` is clean. If it's local-only, `git+file:///home/patrikpersson/dotfiles` works but is fragile and host-specific. Pushing to GitHub first is the right move.

2. **What's the iteration loop for editing dotfiles on the T14?** Currently, editing `~/dotfiles/claude/CLAUDE.md` doesn't update `~/.claude/CLAUDE.md` until you commit + push + flake-update + rebuild. That's a lot of friction for tweaking prompts. Two ways to handle:
   - Accept it: forces clean commits, no "wip" state in the HM symlink
   - Use `git+file://` against the local checkout: faster, but you still need a commit (or `--no-write-lock-file` + `--impure`)
   - Bypass HM for stuff under active iteration: write directly to `~/.claude/CLAUDE.md` and reconcile periodically

3. **What about things HM has a module for** (git, ghostty, ssh, firefox)? The honest answer is: if you're on Option B, HM modules win on NixOS and the dotfiles versions of those specific files are macOS-only. That means accepting some duplication for those — Nix attrs in the HM module on the Linux side, text file in dotfiles on the Mac side. Option A is what eliminates that.

## Decision: split `~/.claude/CLAUDE.md` per host

The "global" CLAUDE.md is loaded into every Claude Code conversation on a machine, so its content is inherently per-machine. Trying to keep one file shared across NixOS and macOS doesn't work — the environment section and the package-manager rule are platform-specific.

Plan (lines up with Option B above):

```
~/dotfiles/
  base/claude/CLAUDE.md                  ← universal: working style, tooling, writing guardrails
  hosts/personal-mac/claude/CLAUDE.md    ← macOS environment + rules + references
  hosts/t14/claude/CLAUDE.md             ← NixOS environment + rules + references
```

Each machine assembles its `~/.claude/CLAUDE.md` by concatenating `base` + its own host slice:
- macOS: `install.sh` does `cat base/claude/CLAUDE.md hosts/personal-mac/claude/CLAUDE.md > ~/.claude/CLAUDE.md`
- NixOS: home-manager does `home.file.".claude/CLAUDE.md".text = builtins.readFile <base> + builtins.readFile <host>`

Current state (2026-05-13): the T14-specific CLAUDE.md lives at `/etc/nixos/home/patrikpersson/claude/CLAUDE.md` — not yet split. The dotfiles version (`~/dotfiles/claude/CLAUDE.md`) still has the original macOS-style content. The split happens as part of the Option B migration; until then, the two diverge intentionally.

## What we did *not* discuss yet but should

- **Secrets** (sops-nix step in roadmap): on macOS, where do secrets live? Same age keys? Different recipient list?
- **Per-host vs. per-platform overrides**: dotfiles already has `hosts/personal-mac/`. The NixOS flake would grow `hosts/t14/` and (eventually) `hosts/<mac>/`. Same idea, different mechanism.
- **Project-level CLAUDE.md** (`/etc/nixos/CLAUDE.md`): per-repo, doesn't need the per-host split machinery. Lives in whichever repo it documents.
